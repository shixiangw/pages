背景与动机

旧方案以 EMQX Rule Engine republish 为核心，上行可用但存在结构性不足：

P1
下行能力缺失
Command、Shadow、Jobs 收远期；Rule republish 难以闭环
P2
Rule 无法承载复杂语义
有状态逻辑 + Schema 查表，Rule SQL 表达力不足
P3
ThingType 转换受限
Rule 无法在运行时引用映射表做字段级转换
P4
ErrNoMatchingSubscribers
DA 与 Fleets topic 不一致，下行阻塞

新方案：独立 Bridge Agent 进程，MQTT 原生 sub/pub + HTTPS ThingDatas API 完成协议适配。

原则

原则说明
Bridge 专职 MQTT所有转换在 Bridge 内完成，不依赖 EMQX Rule republish
Fleets 预配照旧Bridge 向 $emqx/ 发布后，Fleets Rule 写 PG / EMQX Tables
DA 主动注册DA IoT 插件是 Thing/ThingType 注册唯一发起方
映射表真相源da_bridge_* 表为运行时真相源；写入统一走 Bridge Admin API，runtime 只读
Fail-safe映射缺失时丢弃+告警，命令 correlation 严格校验
独立 MQTT 账号Bridge 使用独立 EMQX 服务账号，不依赖 REST API
QoS 1 + MQTT 5 NoLocal关键上下行 topic 统一 QoS 1；MQTT 5 启用 NoLocal 避免自发布回环
Source-of-Truth 在 DB只读 Schema 缓存，权威来源在 PG，支持失效/热重载

阶段行为

MQTT 接入

MQTT 原生客户端

Subscribe + 全部 Publish（Shadow/Event/Command/Jobs 下行）统一走 MQTT client

HTTPS ThingDatas API

Jobs proxy 模式下的 start-next、update 请求走 HTTPS，无需订阅响应 topic

DA_BRIDGE_MQTT_URL=mqtt://localhost:1883
DA_BRIDGE_MQTT_VERSION=5             # 强烈建议 MQTT 5
DA_BRIDGE_MQTT_CLIENT_ID=da-fleets-bridge-{instanceId}
DA_BRIDGE_MQTT_CLEAN_SESSION=false   # 持久会话
DA_BRIDGE_MQTT_QOS=1                 # 关键 topic 统一 QoS 1
DA_BRIDGE_MQTT_NO_LOCAL=true         # MQTT 5：不接收自己发布的消息
DA_BRIDGE_MQTT_RECONNECT_BACKOFF=1s,5s,15s,30s,60s
EMQX REST API 不再作为发布通道，降低依赖。MQTT 3.1.1 无法使用 NoLocal，必须依赖 S5 handler 的 state.desired 过滤。

消息处理流水线

MQTT OnMessage → Topic Router
  → 解析 {productId(versioned), deviceId, thingName, executionId, jobId}
  → Mapping Lookup
    da_bridge_mappings WHERE da_product_id=versionedPid AND da_device_id=?
    miss → 丢弃 + metric + warn log
  → Per-Thing Advisory Lock (变更类消息)
    pg_advisory_lock(HashToInt64("da-bridge-" + thingName))
  → Schema Lookup
    da_bridge_mappings.da_base_product_id + schema_version
    → da_bridge_schema_mappings (mapping_spec)
  → Translator (payload 转换 + 校验)
  → Publish (MQTT client QoS 1 / HTTPS ThingDatas)
  → 结构化日志 + trace_id + latency histogram

路由关键点

DA MQTT topics 使用 versioned productId（如 smart-fan-v1-1）。Bridge 通过通配符 + 匹配并提取该值，在 da_bridge_mappings 中按 da_product_id（versioned）做第一次路由，再用 da_base_product_id 查 schema。

Command Response

S3 — Bridge 查 in-flight 命令表（PG 持久化）：

字段说明
execution_idFleets execution UUID
da_product_idversioned productId（topic 重建）
da_request_id= executionId（直接映射）
expires_atTTL，过期发布 FAILED
 
DA:  {"requestId":"550e8400-...","code":0,"data":{"estimatedDowntime":45}}
OUT: $emqx/commands/…/executions/{id}/response (QoS 1)
     {"status":"SUCCEEDED","result":{"estimatedDowntime":45},"timestamp":...}
幂等性：in-flight 表以 execution_id 为唯一键，使用 ON CONFLICT DO NOTHING 防止重启后重复下发。TTL goroutine 每 30s 扫描过期条目 → 发布 FAILED 响应到 Fleets。

Command 下行

S4 $emqx/commands/things/da-+/executions/+/request

Fleets JobService 发布
{"action":"setInterval","params":{"interval":30},"ttl":60}
Bridge 发布到 DA
{"cmd":"setInterval","params":{"interval":30}}

Bridge 步骤：解析 thingName/executionId → 获取 per-thing advisory lock → 查 mappings → 查 mapping_spec → 幂等写入 in-flight（execution_id 唯一键）→ 发布 DA topic（QoS 1）

✅ 解决 ErrNoMatchingSubscribers：Bridge 订阅存在，Fleets publish 时匹配。
