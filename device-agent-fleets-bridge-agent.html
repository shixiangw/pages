◆ Bridge Agent / Device Agent × Fleets 集成方案
v1.1 评审优化 2026-07-02
Device Agent × Fleets
独立 Bridge Agent 进程双向协议适配 · 替代 EMQX Rule republish
前置：旧方案 v2.5

背景与动机

旧方案以 EMQX Rule Engine republish 为核心，适合上行但存在结构性不足：

P1
下行能力缺失
Command、Shadow、Jobs 被收入远期 TODO，Rule republish 难以闭环
P2
Rule 无法承载复杂语义
Shadow desired、Jobs 状态机需有状态逻辑 + Schema 查表
P3
ThingType 转换受限
Rule 无法在运行时引用映射表做字段级转换
P4
ErrNoMatchingSubscribers
Fleets 原生设备订阅与 DA 不一致，下行阻塞

设计目标

双向互通

上行 + 下行（command/shadow/jobs）一期设计到位

标准路径对齐

Fleets 预配 EMQX Rule → ThingDatas API 路径不变

Schema 可治理

映射持久化、版本化，支持人工修订与 API 校验

最小侵入

Fleets 主进程仅 Migration + 编排 Sidecar

可选启用

Bridge Agent 与 DA 插件均可独立开关

可观测

结构化日志、Prometheus 指标、健康检查、trace_id

原则

原则说明
Bridge 专职 MQTT所有 DA↔Fleets 转换在 Bridge 内完成，不依赖 EMQX Rule republish
Fleets 预配照旧Bridge 向 $emqx/ 发布后，Fleets 既有 Rule 写 PG / EMQX Tables
DA 主动注册DA IoT 插件是 Fleets Thing/ThingType 注册的唯一发起方
映射表为真相源Bridge 独写 da_bridge_* 表，版本与 Product 修订联动
Fail-safe映射缺失时丢弃并告警，不伪造 payload；命令 correlation 严格校验

生命周期

阶段行为
StartDB migrate → 抢 advisory lock → 加载 Schema 缓存 → MQTT 连接（cleanSession=false）→ 批量 subscribe → 注册 handler → in-flight TTL 清理 goroutine
Running处理 MQTT 消息；定期增量同步 Schema；清理过期 in-flight 命令并发布 FAILED 响应
DrainingSIGTERM：停止处理 → 等待 in-flight 完成（超时 SHUTDOWN_TIMEOUT）→ 断开 MQTT → 关闭 DB → 释放锁
Ready probeMQTT 已连接 + 全部 topic 已 subscribe + DB 可达 + advisory lock 持有
Live probe进程存活 + MQTT reconnecting 算 live

MQTT 接入

MQTT 原生客户端

Subscribe + 上行高吞吐 Publish。EMQX REST API 不支持持久订阅。

EMQX REST API v5 /publish

命令/Jobs 响应 publish。复用 Fleets EMQX_API_KEY。

DA_BRIDGE_MQTT_URL=mqtt://localhost:1883
DA_BRIDGE_MQTT_CLIENT_ID=da-fleets-bridge-{instanceId}
DA_BRIDGE_MQTT_CLEAN_SESSION=false   # 持久会话，重启不丢消息
DA_BRIDGE_MQTT_RECONNECT_BACKOFF=1s,5s,15s,30s,60s
cleanSession=false + 持久会话保证重启期间消息不丢。in-flight 命令表在 PG 中持久化。

消息处理流水线

MQTT OnMessage → Topic Router → Mapping Lookup → Schema Registry
    → Translator (payload 转换 + 校验)
    → Publish (MQTT client / EMQX REST API)
    → 结构化日志 + trace_id + latency histogram

映射表

da_bridge_mappings

设备实例映射。键：(da_product_id, da_device_id) → thing_id, thing_name, thing_type_id, schema_version, mqtt_client_id

da_bridge_schema_mappings

Product ↔ ThingType 映射规则。status: draft / active / deprecated。含 mapping_spec JSONB

mapping_spec 结构

{
  "properties": {
    "reportIntervalSec": {
      "writable": true,
      "desiredApply": { "strategy": "command", "command": "setInterval",
        "paramMap": { "reportIntervalSec": "interval" } }
    }
  },
  "events": { "overheat": { "fleetsSeverity": "warn", ... } },
  "commands": { "setInterval": { "daCmd": "setInterval", "inputMap": {...} } },
  "jobs": { "mode": "proxy", "operationCommandMap": {} }
}

desiredApply 策略

strategy行为
command将 desired 中 writable 字段变化映射为一条 DA commands 消息
stateOnly仅更新 Fleets reported，不执行下行
ignore跳过该字段

默认自动映射规则

DA DeviceSpecFleets ThingType默认策略
properties.*schema.properties.*同名映射；用户标记 writable
events.*schema.events.info.*默认 severity=info
commands.*schema.commands.* (sync)parameters → input

Command Response

S3 device-agent/{productId}/device/{deviceId}/responses

Bridge 查 in-flight 命令表（PG 持久化），按 requestId 关联：

DA:  {"requestId":"550e8400-...","code":0,"message":"ok","data":{"estimatedDowntime":45}}
     → code===0 → SUCCEEDED 否则 FAILED
OUT: $emqx/commands/things/{thingName}/executions/{executionId}/response
     {"status":"SUCCEEDED","result":{"estimatedDowntime":45},"timestamp":1715432405}
in-flight 字段说明
execution_idFleets command execution UUID
da_request_id下发 DA 时使用的 requestId (=executionId)
thing_name桥接 Thing 名
expires_atTTL，过期后台清理并发布 FAILED 响应
TTL 清理 goroutine 每 30s 扫描过期条目 → 发布 FAILED 响应到 Fleets → 删除条目。

Command 下行

S4 $emqx/commands/things/da-+/executions/+/request

Fleets JobService 发布
{"action":"setInterval","params":{"interval":30},"ttl":60}
Bridge 发布到 DA
{"cmd":"setInterval","params":{"interval":30}}

Bridge 处理步骤：

  1. 从 topic 解析 thingName、executionId
  2. 查 da_bridge_mappings 得 productId、deviceId
  3. 查 mapping_spec.commands[action] 得 daCmd、inputMap
  4. requestId = executionId（直接映射）
  5. 写入 in-flight 表 → 发布 DA commands topic

✅ 解决 ErrNoMatchingSubscribers：Bridge 作为订阅者，Fleets publish 时 Broker 存在匹配订阅。

Sequence

FleetsUI → FleetsAPI → JobService ──pub req──→ EMQX ──Bridge sub──→ in-flight write
    → pub DA cmd → DA Device execute → response → Bridge → pub to EMQX
    → JobService UpdateExecution → SUCCEEDED

Shadow desired / delta 下行

S5/S6 — DA 无 Shadow 协议，Bridge 将 Fleets Shadow 语义映射为 DA 命令。

Fleets desired: {"reportIntervalSec":60,"powerMode":"eco"}
  → desiredApply[reportIntervalSec] → cmd=setInterval, params={interval:60}
  → desiredApply[powerMode] → cmd=setPowerMode, params={mode:"eco"}
合并策略行为
perField（默认）每个字段独立发一条 DA 命令，最安全
singleCommand所有同 cmd 的字段合并为一条命令的参数

NTP

DA 设备使用 device-agent/{pid}/device/{did}/ntp/request|response。Fleets 无对应标准 topic。Bridge 不处理 NTP。

Jobs 代理模式

commandMap（推荐）

jobDocument 有固定 operation（如 firmware_update）。Bridge 订阅 notify → 映射为 DA 命令 → 响应后回写。

DA 被动收命令，Phase 4 默认模式。

proxy

DA 完全无 Jobs 意识。Bridge 通过 HTTPS ThingDatas API 完整代跑 Jobs 状态机。

DA 不参与，多步进度上报场景。

commandMap 流程

S7/S8 notify → 解析 QUEUED 列表
  → POST /api/v1/thing-datas/jobs/start-next (HTTPS)
  → jobDocument.operation → operationCommandMap → daCmd
  → Publish device-agent/{pid}/device/{did}/commands (MQTT)
  → DA response → POST jobs/{jobId}/update (HTTPS)
