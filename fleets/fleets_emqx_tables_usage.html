Fleets × EMQX Tables 功能使用清单

全面梳理 Fleets IoT 平台对 EMQX Tables (GreptimeDB) 时序数据库的功能使用——表结构、写入路径、查询场景、连接管理与配置

🎯 Fleets 对 EMQX Tables 的依赖定位

EMQX Tables 是 Fleets 的时序数据存储层，承担设备事件历史、影子上报时间线和可选自定义遥测的写入与查询。Fleets 的元数据（设备、影子最新状态、作业等）仍在 PostgreSQL 中。

📊 功能矩阵总览

功能域EMQX Tables 角色涉及表关键组件
设备事件 存储事件历史，支持按时间/类型/严重级别查询 fleets_events TSDBRepo.InsertEvent / QueryEvents
影子上报历史 记录每次 reported 状态的时间线 fleets_shadow_reported TSDBRepo.InsertShadowReported
设备查询预过滤 利用 shadow 时序窗口缩小设备候选集 fleets_shadow_reported FindThingNamesByShadowReportedWindow
自定义遥测 可选：存储设备自定义 MQTT 遥测数据（由 EMQX 规则直接写入） fleets_telemetry EMQX 规则引擎（Fleets 不直接写入）
健康检查 /health 端点报告 Tables 连接状态 — TSDBConnectionManager.Ping
⚡ 关键设计决策：EMQX Tables 不可用时，Fleets 仍可正常启动和运行（优雅降级）。设备查询时序预过滤回退到 PostgreSQL thing_shadows.updated_at；影子写入会失败（因为历史记录不可缺少）；事件写入仅记录 WARN 日志，不阻断请求。

🗂️ 三张时序表

1. fleets_events — 设备事件表 核心

列名类型说明
thing_nameSTRING设备名称
event_typeSTRING事件类型（自定义字符串）
severitySTRING严重级别
tsTIMESTAMP事件时间戳（TIME INDEX）
data_jsonSTRING事件附加数据（JSON 字符串）

PRIMARY KEY: (thing_name, event_type, severity) | TIME INDEX: ts

2. fleets_shadow_reported — 影子上报历史表 核心

列名类型说明
thing_nameSTRING设备名称
tsTIMESTAMP上报时间戳（TIME INDEX）
reportedSTRING上报的 JSON 数据（扁平 key-value）

PRIMARY KEY: thing_name | TIME INDEX: ts | 每个 thing 仅保留最新行（PRIMARY KEY 覆盖）

3. fleets_telemetry — 自定义遥测表 可选

列名类型说明
thing_nameSTRING设备名称
field_nameSTRING遥测字段名
tsTIMESTAMP遥测时间戳（TIME INDEX）
valueDOUBLE遥测数值

PRIMARY KEY: (thing_name, field_name) | TIME INDEX: ts

注意：此表不被 Fleets HTTP API 直接写入。数据由 EMQX 规则引擎通过 InfluxDB 线协议直接写入 GreptimeDB（Fleets 启动时通过 EMQXBrokerSetupService 配置规则）。Fleets 仅负责创建表 Schema。

🔧 Schema 管理细节

Fleets 在启动和热重载时通过 EnsureGreptimeSchema() 管理时序表 Schema，具体步骤：

  1. CREATE TABLE IF NOT EXISTS 创建三张表
  2. 列重命名迁移：ALTER TABLE ... RENAME COLUMN thing_id TO thing_name（旧版本兼容，幂等——列已存在或从未存在时静默跳过）
  3. 类型对齐：ALTER TABLE fleets_events MODIFY COLUMN data_json STRING
    ALTER TABLE fleets_shadow_reported MODIFY COLUMN reported STRING（确保与 EMQX 原生连接器的 InfluxDB 线协议写入类型一致）

📥 两条写入通道

通道 A（主路径，>80% TPS）：EMQX Broker 规则引擎 → GreptimeDB 原生连接器（gRPC 端口 4001）→ InfluxDB 线协议写入 → fleets_events / fleets_shadow_reported
通道 B（辅路径，10-20% TPS）：设备 HTTPS → Fleets ThingDatas API → pgx 驱动（PG Wire 端口 4003）→ SQL INSERT → fleets_events / fleets_shadow_reported

📝 具体写入场景

场景 1：MQTT 设备事件 → fleets_events（通道 A）

设备 MQTT Publish→ $emqx/things/{name}/events/{type}→ EMQX 规则 fleets_events→ GreptimeDB 原生动作→ gRPC 写入 fleets_events

InfluxDB 线协议：fleets_events,thing_name=${thing_name},event_type=${event_type},severity=${severity} data_json=${data_json} ${ts}

场景 2：HTTPS 设备事件 → fleets_events（通道 B）

设备 HTTPS POST→ /api/v1/thing-datas→ ThingDataService.WriteEvent()→ pgx INSERT 写入 fleets_events

失败时仅 WARN 日志，不中断请求（事件为非关键时序数据）

场景 3：MQTT 影子上报 → fleets_shadow_reported（通道 A）+ PostgreSQL（双写）

设备 MQTT Publish→ $emqx/things/{name}/shadow/update→ EMQX 规则（两个动作并行）
动作 1：GreptimeDB 原生
fleets_shadow_reported_to_greptime
→ 写入 fleets_shadow_reported（时序历史）
动作 2：HTTP 回调
fleets_shadow_reported_to_api
→ POST /api/v1/thing-datas/shadow/reported/pg
→ 写入 PostgreSQL（最新状态）

场景 4：HTTPS 影子上报 → 双写（通道 B）

设备 HTTPS POST→ /api/v1/thing-datas→ UpdateShadowReported()
步骤 1：TSDB 先写
InsertShadowReported → fleets_shadow_reported
⚠ 失败则整体操作中止
步骤 2：PG 后写
applyShadowReportedPG → thing_shadows
更新最新状态 + 计算 delta

⚠️ 错误处理策略对比

数据类型写入失败策略原因
事件WARN 日志，不中断请求非关键时序数据，不影响设备核心操作
影子上报ERROR + 返回失败历史记录不可缺少，影响设备查询时窗预过滤

📤 三大查询场景

场景 1：设备事件查询 fleets_events

API: GET /api/v1/things/{name}/events

过滤维度SQL 条件说明
设备名thing_name = ANY($1::text[])支持多设备批量查询
时间范围ts >= $N / ts <= $NRFC 3339 格式
事件类型event_type = $N精确匹配
严重级别severity = $N精确匹配
分页LIMIT N OFFSET M⚠ 内联整数值
GreptimeDB PG Wire 限制：LIMIT / OFFSET 不能使用绑定参数（OID 705 编码问题），必须内联整数值。因此 Fleets 在 Go 端先做范围限定（max 100），再拼入 SQL 字符串。
查询流程：先执行 SELECT count(*) 获取总数，再执行数据查询 SELECT ... ORDER BY ts DESC LIMIT/OFFSET。默认每页 20 条，最大 100 条。

场景 2：设备查询时序预过滤 fleets_shadow_reported

API: POST /api/v1/device-query（Device Query API）

当设备查询中包含时序窗口条件时（tsFrom / tsTo / tsFieldName），Fleets 先查 fleets_shadow_reported 获取候选 thing 列表，再与 PostgreSQL 元数据查询结果取交集：

子场景 2a：任意上报窗口（无 tsFieldName） 
SELECT DISTINCT thing_name FROM fleets_shadow_reported
WHERE ts >= $1 AND ts <= $2 [AND thing_name = ANY($3::text[])]
ORDER BY thing_name LIMIT 1000
子场景 2b：指定字段窗口（有 tsFieldName） 
SELECT thing_name, reported FROM fleets_shadow_reported
WHERE ts >= $1 AND ts <= $2 [AND thing_name = ANY($3::text[])]
ORDER BY thing_name, ts LIMIT 100000

查询结果在 Go 端过滤——只保留 reported JSON 中包含指定 key 的 thing_name。

响应头透传：API 响应中包含 X-TSDB-Candidate-Limit（预过滤上限）和 X-TSDB-Candidate-Truncated（达到上限时为 "true"），帮助客户端了解时序预过滤是否被截断。

场景 3：定时任务（Saved Query）时序预过滤

JobService 执行保存的设备查询时，同样利用 fleets_shadow_reported 进行时序窗口预过滤，机制与设备查询一致。当 TSDB 不可用时回退到 PostgreSQL thing_shadows.updated_at 时间窗口过滤。

🔌 连接方式

Fleets 通过 PostgreSQL Wire Protocol（端口 4003）连接 EMQX Tables / GreptimeDB，使用 pgx v5 驱动。注意：GreptimeDB 也暴露 gRPC 端口 4001 供 EMQX Broker 原生连接器使用。

🔄 热重载机制

TSDBConnectionManager 管理运行时连接池，支持零停机配置热重载：

  1. ConfigRefreshService 每隔 CONFIG_REFRESH_INTERVAL_SECONDS（默认 10 秒）检查 system_configs 表中的 TSDB 配置变更
  2. 检测到变更 → TSDBSettingsService.ReloadFromStore() 重新加载
  3. TSDBConnectionManager.Reload() → 创建新连接池 + EnsureGreptimeSchema + 原子切换 → 旧池异步关闭
  4. 配置指纹（SHA-256）用于判断是否实际发生变更，避免不必要的重连

🏥 健康检查

// db.go — PostgreSQL vs TSDB 健康检查差异
// PostgreSQL:
pool.Ping(ctx)

// EMQX Tables (GreptimeDB):
pool.QueryRow(ctx, "SELECT 1").Scan(new(int))
// Ping() 在 GreptimeDB 上可能返回 "empty statements" 错误

健康检查端点 GET /health 返回 "Tables" 字段，包含 endpoint、status、error（如有）。

🔗 EMQX Broker 连接器配置

POST /api/v1/settings/emqx/setup API 在 EMQX Broker 中自动创建：

资源类型ID用途

Actiongreptimedb:fleets_events_to_greptime事件写入动作（InfluxDB 线协议）
Actiongreptimedb:fleets_shadow_reported_to_greptime影子写入动作（InfluxDB 线协议）
Rulefleets_eventsMQTT 主题 → greptimedb 动作
Rulefleets_shadow_reportedMQTT 主题 → greptimedb 动作 + HTTP 回调
遗留清理：setup 时自动删除旧版 pgsql:fleets_greptimedb_connector 和 http:fleets_greptimedb_sql_connector 连接器（历史兼容）。

🚀 启动流程

  1. 加载环境变量 TSDB 配置 → config.TSDBConfig
  2. 创建 TSDBSettingsService（env + DB 存储回退）
  3. 创建 TSDBConnectionManager
  4. 调用 Reload() → 连接 GreptimeDB → SELECT 1 验证
  5. EnsureGreptimeSchema() → 创建/迁移 3 张时序表
  6. 失败则 WARN 日志："EMQX Tables not available, device query time-series disabled"——应用继续运行
