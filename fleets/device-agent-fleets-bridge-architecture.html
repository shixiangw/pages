Device Agent × Fleets 集成架构

Bridge Agent 独立进程方案 · v1.5 · 2026-07-02

Bridge Agent 作为独立 Sidecar 进程，通过 MQTT 原生 subscribe/publish 双向接入 Broker，完成 DA↔Fleets 协议适配

核心数据流一览

方向 DA 侧 Bridge 动作 Fleets 侧
状态上行 ↑ v1/.../telemetry (type=state) 转 Shadow reported 预配 Rule → ThingDatas
事件上行 ↑ v1/.../event 转 Fleets event 预配 Rule → EMQX Tables
命令响应 ↑ device-agent/.../responses 转 command response 预配 Rule → ThingDatas
命令下行 ↓ device-agent/.../commands 订阅 request → 发布 DA JobService EMQX publish
Shadow 下行 ↓ DA commands（合成） desired/delta → DA 命令 ShadowService EMQX publish
Jobs 下行 ↓ DA commands / 代理请求 notify → commandMap / proxy JobService EMQX publish
Jobs 上行 ↑ DA responses / Bridge 代发 转 jobs/{id}/update 预配 Rule → ThingDatas
Lifecycle EMQX connect/disconnect — 不经 Bridge 预配 Rule → ThingDatas

实施路线图

Phase 1 Bridge 骨架 + 上行 2–3 周
  • 仓库初始化 + 项目骨架 + 生命周期
  • MQTT client（MQTT 5 + NoLocal）+ unified topic router
  • da_bridge_mappings + da_bridge_schema_mappings migration
  • Telemetry / Event translator
  • docker-compose 集成 · E2E 验证
Phase 2 Command 双向 2–3 周
  • In-flight 命令表（PG）+ TTL 清理 goroutine
  • Command 下行 + response 上行 translator
  • Command correlation（executionId ↔ requestId）
  • Bridge Admin API（schema-mappings CRUD）
  • E2E: Fleets POST /commands → DA 执行 → SUCCEEDED
Phase 3 Shadow 下行 1–2 周
  • Per-property desiredApply translator
  • SHADOW_DESIRED_BATCH_MODE（perField / singleCommand）
  • S5 自发布过滤（state.desired 存在性检查）
  • E2E: 控制台改 desired → DA 收到命令 → reported 更新
Phase 4 Jobs 代理 2 周
  • jobs.mode=commandMap：notify → start-next → DA 命令 → update
  • jobs.mode=proxy：ThingDatas HTTPS API 代调用
  • 文档与 UI 能力标注
  • E2E: Fleets 创建 Job → Bridge 代理 → DA 执行 → Job SUCCEEDED
Phase 5 生产化
  • HA Active-Standby（advisory lock + MQTT failover）
  • 监控告警 · 映射 resync Admin 完整实现
  • 性能压测 · 升级回滚文档
