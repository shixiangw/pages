Fleets vs AWS IoT Core & AWS IoT Device Management — 功能对比清单

版本: 2.0 | 生成日期: 2026-06-02<br><br>数据来源:<br>- Fleets: internal/handler/ (32 handlers), internal/service/ (18 services), internal/repository/ (11 repos), internal/model/ (50+ 模型类型), internal/devicequery/ (完整 DSL), internal/router/router.go, migrations/, docs/, README.md, AGENTS.md<br>- AWS: docs.aws.amazon.com/iot/latest/developerguide + aws.amazon.com/iot-core/features + aws.amazon.com/iot-device-management/features

Fleets 复用了 AWS IoT Core MQTT 系统主题的命名惯例（$aws/ → $emqx/），但 不是 API 兼容实现。本文档提供两者在协议、API、数据模型和功能域上的完整对比。

▼ Fleets 优势领域

领域说明
设备模型ThingType 支持丰富 JSON Schema（properties 含类型/单位/范围、events、commands、connectivity、security、lifecycle、edge、relations）。AWS IoT Core 仅支持 searchableAttributes（字符串数组）。Fleets 还支持 Schema 继承（Thing ← ThingType）和 JSONB 任意结构属性。
时序数据库内置EMQX Tables（原 GreptimeDB）作为内置时序数据库，通过 PG 协议（端口 4003）读写。设备数据（事件 fleets_events、影子历史 fleets_shadow_reported、自定义遥测 fleets_telemetry）天然时序化。AWS 无内置时序能力，需外接 Timestream/S3。
设备侧 HTTPS 协议ThingDatas API 提供 9 个设备端 HTTPS 端点（events、shadow reported/read、command response、lifecycle、jobs get/start-next/update），覆盖设备与云端的所有交互场景（双向：设备→云发布、云→设备读取+响应）。AWS IoT Core 设备侧 HTTPS 仅支持单向 publish（POST /topics/{topic}），不支持 影子读取、Jobs 查询、命令响应等需要订阅/响应的操作；设备若要通过 HTTPS 完成完整交互，必须自建轮询或额外部署应用层协议。
标签系统独立的双类型标签实体（Thing 标签 type=0、ThingType 标签 type=1），支持 CRUD、批量创建、自动创建、孤儿清理。标签驱动动态 Group 成员（tagFilter AND 匹配）、搜索筛选。AWS 无独立标签系统，仅以 Attribute 字符串替代。
EMQX Broker 深度集成通过 EMQX V5 REST API 实现下行发布（影子、jobs、commands、事件）；规则引擎 Connector/Action/Rule 可动态配置；回调 API key 自动轮换；no_matching_subscribers 降噪检测；双重发布验证（status + 探针）。
运维管理Teleport SSH 集群注册/续期/停止/重置/状态查询；TLS 证书通过 API 热重载；结构化日志（slog）；请求 ID 追踪（X-Request-ID）；CORS 支持；自动 OpenAPI 3.1 文档。
HA 与配置热重载PostgreSQL Advisory Lock（per-thing 状态变更）；ConfigRefreshService 定期脏检查同步 SignKey / API Key 权限 / TSDB 连接；实例 ID 日志标识。
Jobs 高级功能完整 rollout 策略（常量/指数）、调度窗口（cron）、中止/超时/重试配置；后台 worker 出站队列 + 通知 + 超时 + continuous 编排；设备侧 HTTPS Jobs 协议。AWS 无设备侧 HTTPS Jobs 端点。
影子收敛自检当 delta 收敛时自动将 Job execution 标记为 SUCCEEDED，减少设备与云端的交互轮次。

▼ AWS 优势领域

领域说明
设备预置（Fleet Provisioning）AWS 提供完整的 JITP（Just-in-Time Provisioning）、JITR（Just-in-Time Registration）、Claim-based、Trusted User 四种预置方案。Fleets 仅支持 API 单设备注册和 JSON 批量导入。
设备安全（Device Defender）AWS IoT Device Defender 提供自动化安全审计（Audit）、行为规则检测（Rules Detect）、ML 异常检测（ML Detect）以及 SNS/CloudWatch 告警。Fleets 无此功能。
设备 SDKAWS 提供全平台官方 SDK（C/JS/Python/Java/嵌入式 C），但 SDK 主要基于 MQTT，HTTPS 场景无 SDK 支持。Fleets 仅提供标准 MQTT/HTTPS 协议，设备端需自行实现协议适配。
OTA 固件升级AWS IoT Device Management OTA Update Job + Software Package Catalog 支持固件分发、数字签名、版本追踪、定向更新。Fleets 无此功能。
Secure TunnelingAWS 支持通过 $aws/things/{name}/tunnels/notify 建立安全远程隧道，支持多并发连接，用于远程调试。Fleets 无此功能。
Fleet Indexing 聚合AWS SearchIndex 支持 count/avg/min/max/sum/percentile/cardinality 聚合查询 + Fleet Metrics 自动推送 CloudWatch。Fleets Device Query 不支持聚合和排序。
全托管 HAAWS IoT Core 自动跨区域、跨 AZ 扩展，无需运维底层基础设施。Fleets 需自行运维 EMQX + PostgreSQL + EMQX Tables 集群。
IoT Policy（IAM）AWS 基于 IAM 策略的细粒度权限控制（iot:Connect/Publish/Subscribe/Receive 独立 action，Topic 级别 Resource ARN），支持策略变量 ${iot:Thing.ThingName}。Fleets 基于 Basic Auth + API Key 路径前缀匹配。
Named ShadowsAWS 支持每个设备创建多个命名影子（$aws/things/{name}/shadow/name/{shadowName}）。Fleets 仅支持单经典影子。

Fleets 在当前版本中更适合以下场景：

  • 需要私有化/自托管部署的 IoT 平台（企业内网、边缘数据中心）
  • 设备模型复杂的 IoT 场景（需丰富 Schema、Schema 继承、批量导入导出）
  • 自带 EMQX Broker 的已有 MQTT 基础设施
  • 需要内置时序数据库，不希望外接 Timestream/InfluxDB 等
  • 需要 HTTPS-only 设备接入（设备侧不支持 MQTT 订阅）
  • 需要标签驱动动态组的灵活设备分组方案
