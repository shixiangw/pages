# Fleets 私有化部署方案 — 资源估算与场景评估

> **版本**: v1.1 | **更新日期**: 2026-06-30 | **基点版本**: EMQX Enterprise 5.10.3 + Fleets (Go)
> **修正**: v1.1 大幅下调 Fleets 资源（实际仅处理 API CRUD + 认证回调，主数据流经规则引擎直接入 DB），细化 LB 资源（区分 L4 透传 vs L7 TLS 卸载）

## 1. 系统架构总览

### 1.1 数据流澄清（关键）

```
设备数据上报（如影子上报、遥测）
    │
    ├─ MQTT → EMQX ──→ GreptimeDB (规则桥接 gRPC, 主路径, 不经过 Fleets)
    │                └─→ Fleets ThingDatas API (规则 HTTP POST, 仅 lifecycle/shadow PG 副本/command response)
    │                                               └─→ PostgreSQL (少量写入)
    │
    └─ HTTPS → Fleets ThingDatas API ──→ GreptimeDB (时序)
                                         └─→ PostgreSQL (非时序)

Fleets 实际承载的流量：
  - 管理 API (设备 CRUD、影子查询、作业/命令管理) — 低 QPS
  - Broker 认证回调 (设备连接/断开时验证) — 极低 QPS
  - 规则引擎 HTTP POST 回调 — 中等 QPS (不包含主时序数据)
  - EMQX API 下行发布 (指令/影子/通知) — 低 QPS
```

**核心结论**: 时序数据（占比 80%+ 的 TPS）流经 `EMQX → GreptimeDB`，不经过 Fleets。Fleets 处理的是剩余的非时序管理面和部分非时序设备面流量，负载远低于原始 TPS。

### 1.2 依赖组件清单

| # | 组件 | 角色 | 数据流 | 关键端口 |
|---|------|------|--------|----------|
| 1 | **EMQX Enterprise** | MQTT Broker — 设备连接、消息路由、规则引擎、认证鉴权 | 设备 ←MQTT/TLS→ EMQX → **GreptimeDB** (规则桥接写入 TS)、EMQX → **Fleets** (规则 HTTP POST 写入 PG) | 1883(MQTT), 8883(MQTTS), 8084(WSS), 18083(API/Dashboard) |
| 2 | **Fleets (Go)** | 业务后端 — 设备注册、影子、作业、命令、认证、管理 API | 设备 ←HTTPS→ Fleets → **PostgreSQL** (元数据)、Fleets → **GreptimeDB** (查询 TS)、Fleets → **EMQX API** (下行发布) | 8080(HTTP), 8443(HTTPS) |
| 3 | **PostgreSQL 16** | 元数据存储 — 设备注册表、影子状态、作业、命令、API 密钥、设备凭证 | 连接数: Fleets 实例数 × 连接池 | 5432 |
| 4 | **GreptimeDB (EMQX Tables)** | 时序数据库 — 事件历史、影子上报历史、可选遥测数据 | EMQX 规则桥接 (gRPC 4001) 写入，Fleets 查询 (PG wire 4003) | 4000(HTTP), 4001(gRPC), 4003(PG wire) |
| 5 | **负载均衡器** | TLS 卸载（可选）、流量分发 | 外部 → LB → EMQX / Fleets | 443(TLS), 8883(MQTTS) |
| 6 | **监控告警** | Prometheus + Grafana / ELK | 各组件暴露 metrics 端点 | — |

### 1.3 网络拓扑

```
                  ┌───────────── LB 层 ───────────────┐
                  │  MQTT: L4 TCP 透传 (haproxy/云 NLB)   │
                  │  HTTP: L7 TLS 卸载 (haproxy/云 ALB)   │
                  │  或 云 LB (NLB for MQTT + ALB for API) │
                  └──────────────────┬──────────────────┘
                                     │
           ┌─────────────────────────┼─────────────────────────┐
           │  MQTT(L4)               │  HTTP(L7)               │
     ┌─────▼─────────┐      ┌───────▼─────────┐      ┌───────▼──────────┐
     │  EMQX Cluster  │      │  Fleets (Go)   │      │  Admin / Ops     │
     │  (N nodes)     │      │  2-3 replicas  │      │  Dashboard       │
     │  (MQTTS/TLS)   │      └───────┬─────────┘      └──────────────────┘
     └─────┬──────────┘              │
           │                         │
     ┌─────┴──────────┐   ┌─────────┴─────────┐
     │  GreptimeDB    │   │  PostgreSQL 16    │
     │  (时序, gRPC)  │   │  (元数据, pgx)    │
     └────────────────┘   └───────────────────┘

数据流说明:
  MQTT 上行:  设备 ─MQTTS→ LB(L4透传) ───→ EMQX ──gRPC──→ GreptimeDB (主)
                                                ─HTTP──→ Fleets → PG (辅: lifecycle/shadow副本)
  HTTPS 上行: 设备 ─HTTPS→ LB(L7卸载) ──→ Fleets ──→ GreptimeDB / PG
  下行:       Fleets ─EMQX API→ EMQX ─MQTT→ 设备
```

## 2. 三个部署规模下的资源估算

### 2.1 估算假设

| 参数 | 假设值 | 说明 |
|------|--------|------|
| 消息 QoS | QoS 1 | 默认生产级可靠性 |
| 消息 Payload | 256 字节 | 影子上报 + 事件，含 JSON 开销 |
| 每连接空闲内存 (TCP) | ~250 KB | EMQX 长连接基础开销 |
| 每连接空闲内存 (TLS) | ~500 KB | TLS 握手 + 会话缓存额外 200-300 KB |
| 每连接空闲内存 (mTLS) | ~600 KB | 双向认证额外开销 |
| 设备数据上报频率 | 每 5 分钟 1 次影子上报 | 典型 IoT 场景 |
| 事件产生频率 | 每设备每天 10 条事件 | 连接/断开/告警等 |
| 数据传输比 | 上行:下行 = 8:2 | 以数据上报为主，指令为辅 |
| 数据保留周期 | 90 天 | GreptimeDB 时序数据保留 |
| **Fleets 实际承载 TPS** | 仅总量的 **10-20%** | 主时序流经 EMQX→GreptimeDB，不经过 Fleets |
| Fleets 高可用副本 | 2-3 副本 | 轻量 Go 服务，2 副本足够，3 副本冗余 |
| PostgreSQL 连接池 | 每个 Fleets 实例 20-40 连接 | pgx 默认连接池大小 |
| LB 工作模式 | **MQTT→L4 TCP 透传** / **API→L7 TLS 卸载** | 见下文详细说明 |

#### Fleets 负载校正（关键）

Fleets **不承载**规则引擎写入 GreptimeDB 的时序数据路径，仅处理：

| Fleets 职责 | 10 万设备 QPS | 50 万设备 QPS | 100 万设备 QPS | 说明 |
|------------|:-------------:|:-------------:|:--------------:|------|
| 管理 API (CRUD) | < 5 | < 20 | < 50 | 管理员操作，非持续流量 |
| Broker 认证回调 | ~0.3 | ~1.7 | ~3.3 | 仅设备重连时触发 |
| 规则 HTTP POST (shadow PG 副本) | ~333 | ~1,667 | ~3,333 | 5min/次上报 |
| 规则 HTTP POST (lifecycle/command) | ~3 | ~17 | ~33 | 事件 + 指令响应 |
| EMQX API 下行发布 | ~55 (10万设备/30min) | ~278 | ~556 | 指令/影子/通知 |
| **Fleets 总处理 QPS** | **~400** | **~2,000** | **~4,000** | Go 轻松应对 |

> Go HTTP 服务器单核可处理 **5,000+ QPS**（简单 JSON CRUD），Fleets 负载在三个规模下均极低。

### 2.5 关键参数依据

#### EMQX 单节点承载能力（参考 EMQX 官方性能测试）

> 数据来源: [EMQX 官方性能参考文档](https://docs.emqx.com/zh/emqx/latest/performance/performance-reference.html)（已验证，数据属实）
> 测试环境: 华为云 c7.xlarge.2 (4 vCPU / 8 GB), EMQX v5.1.6 开源版, 128 字节 Payload, 1000 发布者 + 1000 订阅者 1:1 通信
>
> **注意**: 官方测试使用 128 字节 Payload，本指南估算采用 256 字节。实际生产中 payload 越大 CPU 开销越高，建议按实测调整。

| QoS 等级 | 推荐负载 (TPS) | CPU 使用率 | 内存使用率 | 平均延迟 | P99 延迟 |
|----------|:-------------:|:----------:|:----------:|:--------:|:--------:|
| QoS 0 | ~57K | 78.1% | 36.2% | 2.07 ms | 98.33 ms |
| QoS 1 | ~40K | 75.5% | 66.8% | 2.35 ms | 69.49 ms |
| QoS 2 | ~24K | 69.0% | 66.4% | 2.02 ms | 58.70 ms |

**结论**: 对 QoS 1 场景，单节点（8 vCPU/16 GB 以上）可承载 **15-25K TPS**。考虑到 TLS 开销和连接数，我们的节点规格按以下原则分配：

- **10 万连接 / 1 万 TPS**: 2 节点 (8 vCPU/16 GB)，CPU 使用率约 40-50%
- **50 万连接 / 2 万 TPS**: 4 节点 (16 vCPU/32 GB)，CPU 使用率约 50-60%
- **100 万连接 / 5 万 TPS**: 6 节点 (16 vCPU/32 GB)，CPU 使用率约 60-70%

#### 每连接内存估算

| 场景 | 每连接内存 | 说明 |
|------|-----------|------|
| TCP (无 TLS) | ~250 KB | 不含会话/遗嘱消息 |
| TLS (单向认证) | ~500 KB | TLS 握手缓存 + 会话状态 |
| TLS (双向认证 mTLS) | ~600 KB | 额外证书链验证状态 |

#### Fleets (Go) 实际负载估算

Fleets 是 **Go HTTP API 服务器**，不参与主数据流。其负载分三部分：

| 流量来源 | 10 万设备 | 50 万设备 | 100 万设备 | 说明 |
|---------|:--------:|:--------:|:----------:|------|
| 管理 API (CRUD) | ~5 QPS | ~20 QPS | ~50 QPS | 极少 |
| 规则引擎 HTTP POST (shadow PG 副本) | ~333 QPS | ~1,667 QPS | ~3,333 QPS | 每设备每 5 分钟 1 次 |
| 规则引擎 HTTP POST (lifecycle/命令/作业) | ~3 QPS | ~17 QPS | ~33 QPS | 事件驱动 |
| Broker 认证回调 | ~0.3 QPS | ~1.7 QPS | ~3.3 QPS | 仅重连时 |
| EMQX API 下行发布 | ~55 QPS | ~278 QPS | ~556 QPS | 影子/指令 |
| **Fleets 总处理** | **~400 QPS** | **~2,000 QPS** | **~4,000 QPS** | |

> Go 1 个 vCPU 即可处理 5K+ QPS 的简单 JSON CRUD。Fleets 在三个规模下 **2-4 vCPU 绰绰有余**。

#### PostgreSQL 估算依据

- 每行 Shadow 约 1-5 KB JSON（含索引）
- Per-thing 约 5 条相关记录 (shadow, auth, commands, jobs, groups)
- **PG 实际写入 TPS 远低于总 TPS**：仅 lifecycle 事件、shadow PG 副本、command response、job status 等非时序数据经过 PG。时序数据（占总 TPS 80%+）经规则引擎直接写入 GreptimeDB
- PG 写入 TPS 估算：10 万设备 ~400 TPS，50 万 ~1,700 TPS，100 万 ~3,400 TPS（均含 shadow PG 副本）
- PG 连接池建议: Fleets 实例数 × 20-40 连接 + EMQX 规则引擎 HTTP 回调连接

#### GreptimeDB 估算依据

- 数据压缩比 **1:8 ~ 1:10**（GreptimeDB 官方数据）
- CPU:内存比例建议 **1:4**
- 对象存储成本远低于本地 NVMe，时序数据强烈建议使用
- 本地磁盘主要用于 WAL + 查询缓存

## 3. 场景可行性评估

### 3.1 评估维度与标准

| 维度 | 评级 | 标准 |
|------|------|------|
| 架构支持度 | ✅ / ⚠️ / ❌ | 当前代码/架构是否原生支持 |
| 实施复杂度 | 低 / 中 / 高 | 需要多少额外开发或配置 |
| 性能风险 | 低 / 中 / 高 | 在该规模下是否存在瓶颈 |

### 3.2 功能场景评估表

| 场景 | 支持度 | 实施复杂度 | 大规模性能风险 | 说明 |
|------|--------|-----------|--------------|------|
| **TLS 单向认证** | ✅ 原生支持 | 低 | 低 | EMQX 原生支持 MQTTS 8883 端口，Fleets 支持通过 API 自动配置 TLS 证书。1-way TLS 只需配服务端证书 |
| **TLS 双向认证 (mTLS)** | ✅ 原生支持 | 低 | 低 | Fleets 通过 `mqttDeviceAuthMode: "mtls"` 原生支持。设备携带 X.509 客户端证书，Fleets 验证 CN、状态和过期时间。证书可通过 Fleets API 批量管理 |
| **设备注册** | ✅ 原生支持 | 低 | 低 | Fleets Thing 注册 API 完备，支持属性、唯一标识、批量导入 |
| **三元组配置查询** | ✅ 原生支持 | 低 | 低 | 设备连接凭据（DeviceName/DeviceSecret/ProductKey 等）存储在 PostgreSQL `thing_credentials` 和 `thing_certs` 表中，通过管理 API 查询 |
| **设备状态监控** | ✅ 原生支持 | 低 | 中 | 基于 EMQX lifecycle 事件 (`$events/client_connected` / `$events/client_disconnected`)。50 万以上连接时 lifecycle 规则可能产生突发写入 |
| **数据上报存储** | ✅ 原生支持 | 低 | 中 | 影子上报 → GreptimeDB (历史) + PostgreSQL (最新快照)。可选高频率遥测通过自定义 MQTT 主题 + 规则引擎写入 GreptimeDB。100 万规模下 TS 写入是瓶颈，推荐对象存储 |
| **指令下发** | ✅ 原生支持 | 低 | 低 | 通过 EMQX V5 REST API (`POST /api/v5/publish`) 下行，路由到 `$emqx/things/{name}/shadow/update` 或命令主题。Fleets 无 MQTT 订阅，无客户端竞争风险 |

### 3.3 详细场景分析

#### 3.3.1 TLS 双向认证（mTLS）

**架构支持**: Fleets 已有完整的 mTLS 支持链：

1. EMQX 监听器配置 MQTTS 8883 端口 + `two_way` TLS 模式
2. Fleets 提供 `/api/v1/settings/emqx/listeners/mqtts/certificates` API 自动生成/配置证书
3. 设备登录时 EMQX 将客户端证书 CN 传给 Fleets broker-auth 回调验证
4. `mqttDeviceAuthMode: "mtls"` 模式下，仅 mTLS 设备被允许连接

**大规模注意事项**:

| 关注点 | 影响分析 |
|--------|---------|
| 证书管理 | 100 万设备 × 唯一证书，Fleets 使用 `thing_certs` 表管理，需做好 CA 层级（中间 CA 签子 CA） |
| TLS 握手开销 | 每连接 ~600 KB 内存，10 万连接约 **60 GB**（纯 TLS 开销），占 EMQX 内存大头 |
| 证书撤销 | OCSP/CRL 机制需额外配置，大规模设备批量撤销需提前规划 |
| 证书轮换 | 建议设备证书 1 年有效期自动轮换 |

**结论**: ✅ **完全支持**。mTLS 在架构层面完整实现，生产部署只需配好 CA 体系和证书生命周期管理。

#### 3.3.2 设备注册与三元组管理

**数据流**:

```
管理端 → POST /api/v1/things (创建设备)
        → 自动生成 mqtt_client_id (或指定)
        → 创建 thing_credentials (username/bcrypt_password)
        → (可选) 创建 thing_certs (X.509)
设备开箱 → 获取三元组 → 连接 EMQX (MQTT) / Fleets (HTTPS)
        → EMQX 回调 Fleets broker-auth 验证
        → 认证通过 → 开始上报/接收指令
```

**大规模瓶颈分析**:

| 操作 | 10万 | 50万 | 100万 |
|------|------|------|-------|
| 单设备注册 | < 50ms | < 50ms | < 50ms |
| 批量注册 (1000/批) | ~2s | ~2s | ~2s |
| 三元组查询 | ~5ms | ~10ms | ~15ms (需索引优化) |
| 认证回调 (QPS) | ~0.3 QPS (每5min/设备) | ~1.7 QPS | ~3.3 QPS |

**结论**: ✅ **完全支持**。认证回调 QPS 极低（设备上报周期远小于连接频次），瓶颈不在认证链。

#### 3.3.3 数据上报与存储

**两种路径**:

| 路径 | 可靠性 | 写入并发 | 适用场景 |
|------|--------|---------|---------|
| Shadow reported (MQTT `$emqx/things/{name}/shadow/update`) | 高 (QoS 1) | PG + TSDB 双写入 | 核心设备状态，需实时查看最新值 |
| 自定义遥测 (用户 topic → 规则引擎 → GreptimeDB) | 中 (无确认) | 纯 TSDB 写入 | 高频传感器数据，不需要实时响应 |

**写入模型分析** (假设每台设备每 5 分钟上报一次，每次 1 KB)：

| 规模 | 影子上报 TPS | 影子 PG 写入 TPS | TSDB 写入 TPS | 日增 TS 原始 |
|------|-------------|-----------------|--------------|-------------|
| 10万 | ~333 | 333 | 333 + 事件 ≈ 360 | ~215 GB |
| 50万 | ~1,667 | 1,667 | 1,667 + 事件 ≈ 1,800 | ~430 GB |
| 100万 | ~3,333 | 3,333 | 3,333 + 事件 ≈ 3,600 | ~1.08 TB |

**结论**: ✅ **完全支持**。

- PG 写入压力很低（单节点 8 核 PG 即可轻松处理 5K+ TPS INSERT）
- GreptimeDB 写入的理论上限约 300K 数据点/秒（8 核 32 GB 典型配置），100 万规模约 3.6K/秒，完全在范围内
- 数据保留建议：全量保留 90 天 → 使用对象存储；热数据（近 7 天）在本地 NVMe

#### 3.3.4 指令下发

**数据流**:

```
管理端 → POST /api/v1/things/{name}/shadow (更新 desired)
Fleets → EMQX V5 API POST /api/v5/publish
       → 设备收到 $emqx/things/{name}/shadow/update
       → 设备回复 shadow/update (reported 同步最新状态)
```

**吞吐分析**:

| 指标 | 估算 | 说明 |
|------|------|------|
| EMQX V5 API 发布性能 | ~100K+ msg/s | EMQX API 设计可承载高并发发布 |
| Fleets EMQX API 调用 | 每次下发 1 次 HTTP 调用 | Go HTTP 客户端 + 连接池 |
| 典型指令下发频率 | 每设备每 30 分钟 1 条 | 10万设备 ≈ 55 TPS，远低于上限 |

**结论**: ✅ **完全支持**。指令下发通过 EMQX V5 REST API，无 MQTT 客户端竞争，架构天然适合高并发。

| PG 查询 | 🟢 低 | 🟡 中 (设备查询 P99) | 🟡 中 |
| GreptimeDB 写入 | 🟢 低 | 🟢 低 | 🟢 低 |

| **LB (TLS 卸载 / 场景 B)** | 🟢 **低** | 🟢 **低** | 🟡 **中 (TLS 握手 ≈0.6 核)** |
| 网络带宽 | 🟢 低 | 🟡 中 | 🔴 高 |
| TLS 握手 (EMQX 端) | 🟢 低 | 🟡 中 | 🟡 中 |

> 🟢 无瓶颈 | 🟡 可接受，建议监控 | 🔴 需重点关注或优化

### 4.2 负载均衡器资源分析

> 你说得对，两个关键点：(1) LB 应采用 **多节点、小规格** 方式部署（天然适合横向扩展）；(2) **TLS 卸载放 LB 做** 和 **不放 LB 做** 是两种场景，资源需求差异大，应该分开算。

#### 场景 A：LB 做 TCP 透传（TLS 在 EMQX 端终结）

LB 只做 IP:Port 转发（L4），不解包、不做 TLS。TLS 加解密负担在 EMQX 节点上。

**资源特征**:
- CPU: 内核态转发，几乎为零（<5% 单核）
- 内存: 主要是 conntrack 连接跟踪表 (~3.5 KB/连接)
- 决定因素: **并发连接数**，而非吞吐量

| 规模 | 总连接数 | conntrack 内存 | 推荐 LB 规格 | 说明 |
|:-----|:-------:|:-------------:|:------------|:----|
| 10 万 | ~100K | ~350 MB | 3 × **2 vCPU / 4 GB** | haproxy mode tcp，每节点 ~33K 连接 |
| 50 万 | ~500K | ~1.75 GB | 4 × **4 vCPU / 8 GB** | 每节点 ~125K 连接 |
| 100 万 | ~1M | ~3.5 GB | 5 × **4 vCPU / 8 GB** | 每节点 ~200K 连接 |

#### 场景 B：LB 做 TLS 卸载（TLS 在 LB 端终结）

LB 终结 MQTTS (8883) 和 HTTPS (443) 的 TLS，将明文 TCP/HTTP 转发给后端。

**资源特征**:
- CPU: 取决于 **TLS 握手速率**（高开销）和 **持续加密流量**（AES-NI 硬件加速，开销极低）
- 内存: 同场景 A (conntrack) + TLS 会话缓存
- 决定因素: **新建连接速率** + 并发连接数

**TLS 握手 CPU 估算**（ECDSA P-256，推荐）:

| 规模 | 假设重连率 | 每秒新握手 | CPU 开销/秒 | 说明 |
|:-----|:---------:|:---------:|:----------:|:----|
| 10 万 | 5%/小时 | ~14 次/秒 | <0.1 核 | IoT 设备长连接为主 |
| 50 万 | 5%/小时 | ~69 次/秒 | ~0.3 核 | 仍有大量余量 |
| 100 万 | 5%/小时 | ~139 次/秒 | ~0.6 核 | 仍不到 1 核 |

> ECDSA 握手每核约 0.1ms，RSA 2048 约 0.5ms。IoT 长连接场景下重连率低，TLS 握手 CPU 不是瓶颈。
> 持续加密流量 ~12.8 MB/s (100万/5万 TPS × 256 byte)，AES-NI 硬件加速下几乎无感知。

| 规模 | 总连接数 | conntrack | TLS 额外 | 推荐 LB 规格 | 说明 |
|:-----|:-------:|:---------:|:--------:|:------------|:----|
| 10 万 | ~100K | ~350 MB | 低 | 3 × **4 vCPU / 8 GB** | 小规格多节点即可 |
| 50 万 | ~500K | ~1.75 GB | 低 | 4 × **8 vCPU / 16 GB** | 为 TLS 握手留余量 |
| 100 万 | ~1M | ~3.5 GB | 低 | 5 × **8 vCPU / 16 GB** | 多节点分摊握手压力 |

> 即使在 TLS 卸载模式下，CPU 瓶颈也不明显，因为 IoT 长连接场景重连率低、单条消息小。如果对端是高频重连场景（如手机 App），则需按实际重连率重新计算。

#### LB 规格总结（两个场景对比）

```
┌──────────────── LB 部署策略 ────────────────┐
│                                              │
│  场景 A (TCP 透传):                           │
│    10 万:  3 × 2 vCPU / 4 GB   (haproxy)     │
│    50 万:  4 × 4 vCPU / 8 GB   (haproxy)     │
│    100 万: 5 × 4 vCPU / 8 GB   (haproxy)     │
│    资源低、简单、推荐                          │
│                                              │
│  场景 B (TLS 卸载):                           │
│    10 万:  3 × 4 vCPU / 8 GB   (haproxy+ssl) │
│    50 万:  4 × 8 vCPU / 16 GB  (haproxy+ssl) │
│    100 万: 5 × 8 vCPU / 16 GB  (haproxy+ssl) │
│    CPU 略高，但可统一管理 TLS 证书              │
│                                              │
│  公有云方案:                                   │
│    场景 A → NLB (Network LB)                 │
```

#### 三种部署路径对比

| | 自建 (TCP 透传) | 自建 (TLS 卸载) | 公有云方案 |
|--|:--------------:|:---------------:|:---------:|
| 运维复杂度 | 中 | 高 (证书管理) | **低** |
| CPU 需求 | 极低 | 中 | 无感知 |
| 证书管理 | EMQX 各自管 | LB 统一管 | ACM 自动管理 |
| 推荐指数 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

> **建议**: 有公有云直接 NLB + ALB/ACM，省心且弹性更好。自建场景优先选 TCP 透传方案，把 TLS 卸载给 EMQX，资源需求最低。如果确实需要 LB 统一管理 TLS 证书，选择场景 B，但节点规格适当上浮。

### 6.2 总资源汇总

**场景 A（推荐）：LB 做 TCP 透传，TLS 卸载在 EMQX**

| 规模 | 总物理节点 | 总 vCPU | 总 RAM | 网络带宽 | 存储 |
|:-----|:---------:|:-------:|:------:|:--------:|:----|

| **100 万 / 5 万 TPS** (TCP 透传) | 18 台 | **464 vCPU** | **1,568 GB** | 8 Gbps+ | 3 TB NVMe + 12 TB 对象存储 |

**场景 B：LB 做 TLS 卸载，TLS 证书由 LB 统一管理**

| 规模 | 总物理节点 | 总 vCPU | 总 RAM | 主要差异 |
|:-----|:---------:|:-------:|:------:|:--------|

### 6.3 建议

1. **Fleets 资源无需高配** — Go 轻量服务，2-4 vCPU / 4-8 GB × 2-3 副本即可覆盖所有规模，主要瓶颈不在 Fleets
2. **LB 选型：先定 TLS 卸载位置，再看部署方式**
   - TLS 卸载在 EMQX（场景 A）：LB 仅需 2-4 vCPU 小规格多节点，CPU 占用率 <5%
   - TLS 卸载在 LB（场景 B）：LB 节点规格上浮一档（IoT 长连接场景 TLS 开销依然可控，但需留余量）
   - 部署方式：自建选 **多节点小规格**（如 5 × 4vCPU 替代 2 × 16vCPU）；公有云直接 NLB + ALB/ACM
3. **时序数据使用对象存储** — GreptimeDB 冷数据建议使用 S3/MinIO，相比本地存储可节省 70%+ 成本
4. **使用 Fleets API 进行 EMQX 配置** — 生产环境使用 `POST /api/v1/settings/emqx/setup`，避免 shell 脚本（shell 脚本使用 HTTP 型 GreptimeDB 动作，不适合生产）
5. **100 万规模部署前通过压测验证** — EMQX 官方建议每个部署规模都经过实际压测来校准资源配置
6. **大连接数下严格调优内核参数** — 文件描述符、TCP 缓冲区、连接跟踪参数必须按 EMQX 官方文档调整
7. **PostgreSQL 配置** — 连接池大小、`max_connections`、`shared_buffers` 按节点规模调优；建议使用 PgBouncer 管理大规模连接
