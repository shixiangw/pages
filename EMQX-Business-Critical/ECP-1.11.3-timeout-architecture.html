关键结论：路径 A (edge_tunnel) 使用的 utils.GetClient() 没有设置 Client.Timeout，因此 X-ECP-Timeout=200 设置的 200s context 超时不会被截断。路径 B (fabricHttpProxy) 的 Client.Timeout=30s 和路径 A 完全独立。

2. 路径 A: edge_tunnel 超时链路详解

MQTT 代理模式：timeoutCtx 同时用于 MQTT Pub Topic、Sub Topic 和响应 Channel 等待。三者共享同一个 timeoutCtx，任一环节超时都会返回 502。

5. 超时优先级关系

路径 A（edge_tunnel）：X-ECP-Timeout=200 → Context deadline = 200s，Client 无 Timeout → 200s 完全生效。
路径 B（http_proxy）：无 X-ECP-Timeout，Client.Timeout = 30s → 30s 硬上限。两条路径互不干扰。

6. 常见问题

Q: X-ECP-Timeout=200 会被 Client.Timeout 截断吗？

A: 不会。edge_tunnel.go 路径使用 utils.GetClient()，该 Client 没有设置 Timeout。超时通过 context.WithTimeout(c.Request.Context(), 200s) 控制，全程 200s 有效。

Q: Client.Timeout=30s 和 Context deadline 冲突时谁生效？

A: 两者取最小值。Go 的 http.Client.Do() 会将 Client.Timeout 也写入 request context，最终 deadline = min(Client.Timeout, context deadline)。路径 A 没有 Client.Timeout，所以 context deadline 唯一生效。

Q: DialContext=5s 是什么意思？

A: TCP 三次握手的超时。只控制从 connect() 调用到连接建立成功的阶段。连接建立后，数据传输超时由上层 context 或 Client.Timeout 控制。因此即使 X-ECP-Timeout=200，TCP 建连仍必须在 5s 内完成，否则报错。

Q: MQTT 代理模式超时如何工作？

A: timeoutCtx 同时用于 Pub Topic 发送、Sub Topic 订阅和 Channel 等待，三者共享同一个 context。200s 的 deadline 从 Tunnel 入口开始计时，任一环节超时返回 502。
