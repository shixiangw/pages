ProviderRuntimeService 架构与原理

多后端 IoT 平台适配层 — Facade 模式下的 Default 与 Fleets 双后端交互全景

1 在 agent-gateway 中的定位

ProviderRuntimeService 是 agent-gateway 与外部 IoT 平台之间的统一抽象层。它同时被 AgentRouter、WorkflowRunner、HTTP API 三个模块消费，根据产品的 iotPlatformProviderId 动态路由到不同的后端实现。

2 Facade 模式：多后端路由原理

ProviderRuntimeService 维护一张 Record<providerId, ProviderBackend> 的 Map，所有公开方法都通过 getBackend(product) 根据产品的 iotPlatformProviderId 动态分发到对应的后端实例。

3 上行数据流：Mirror Ingest → WorkflowRunner

Fleets Backend 创建独立的 MQTT 连接订阅 $emqx/things/+/shadow/update、$emqx/things/+/events/+ 等 topic。设备属性/事件通过 DeviceStore Adapter 写入 SQLite，并通过 onRealtimeEvent 回调驱动 WorkflowRunner。

4 下行数据流：命令执行与操作

网关通过 ProviderRuntimeService 执行设备的命令、影子更新、巡检、注册等操作。Fleets 后端走云端 HTTP API，Default 后端走本地 MQTT。

5 supportsTool 与 Backend 方法映射

只有 4 个 ProviderRuntimeService 方法暴露为 AI tool，对应的 capability 检查已放在 AgentRouter 构建 tools 集的逻辑中。其余方法只由内部模块调用，不会出现在 AI 的 function call 列表中。
