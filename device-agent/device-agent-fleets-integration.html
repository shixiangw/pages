🧩 Device Agent + Fleets 集成方案

Device Agent 为上层的 IoT 智能应用层，Fleets 为底层设备管理平台。
集成核心在 Device Agent 端：生成的设备 SDK 直接满足 Fleets 接入规范，DA 的现有功能直接适配 Fleets 协议控制设备。

📦 DA 设备 SDK → Fleets 规范 🤖 DA AI 功能适配 Fleets 协议 🔗 DA 经 Fleets API 控制设备 🎮 统一经 Fleets 下发

🎯 定位与边界

核心定位： Device Agent 是 IoT 智能应用层，Fleets 是设备管理层。两者是互补关系，不是替代关系。 DA 利用 Fleets 的 ThingType Schema 生成兼容的设备 SDK，同时 DA 的 AI 功能经 Fleets API 与设备通信（不再直连 MQTT）。

📱 Device Agent 负责

  • • AI Agent 会话式设备控制
  • • 生成设备 SDK（TS / C / Python）
  • • NL 语义理解 → 设备命令映射
  • • 设备数据分析与洞察
  • • 策略规则引擎
  • • Web 控制台与可视化

🏭 Fleets 负责

  • • 设备注册与鉴权（Thing Registry）
  • • 物模型定义（ThingType Schema）
  • • Device Shadow 三态管理
  • • 作业调度（Jobs）
  • • 命令派发（Commands）
  • • 设备查询与分组
集成原则 — 三不碰： ① DA 不碰 Fleets 的设备注册、鉴权、Shadow 持久化； ② Fleets 不碰 DA 的 AI 会话、SDK 生成、NL 控制； ③ 设备不感知 谁在下发指令 —— 所有下行经 Fleets 统一经 $emqx/ 主题送达。

🏛️ 整体架构

设备通过 EMQX Broker 与 Fleets 通信（Fleets 代管 MQTT 通道）。DA 作为 AI 智能层，统一经 Fleets API 与设备交互（读取状态、下发控制），不再直连 EMQX。

🏗️ Device Agent 平台技术架构

更宏观的 Device Agent 平台视角：边缘设备通过 Device Agent SDK 接入平台，平台侧由 EMQX、Device Agent（AI 能力）和 Device Management（数据管理）组成，同时支持第三方 Agent 通过 A2A 协议互通。

📦 设备 SDK 生成 · 直接满足 Fleets 接入规范

核心能力： Device Agent 的设备 SDK 生成器以 Fleets ThingType Schema 为输入，生成原生支持 $emqx/ 主题和 Fleets 三态 Shadow 协议的设备 SDK。设备开发者无需理解 Fleets 协议细节。

🔧 DA 生成的设备 SDK（输出）

class ThermostatDevice {
  private client: MqttClient;
  private thingName: string;

  constructor(thingName, brokerUrl, creds) {
    this.thingName = thingName;
    this.client = connect(brokerUrl, creds);
  }

  // === Shadow 同步 ===
  async reportState(state) {
    await this.client.publish(
      `$emqx/things/${this.thingName}/shadow/update`,
      { state: { reported: state } }, { qos: 1 }
    );
  }

  // === 事件上报 ===
  async emitEvent(severity, eventType, data?) {
    await this.client.publish(
      `$emqx/things/${this.thingName}/events/${eventType}`,
      { eventType, severity, data, timestamp: Date.now() }
    );
  }

  // === 命令处理 ===
  onCommand(handler) {
    this.client.subscribe(
      `$emqx/commands/things/${this.thingName}/executions/+/request`,
      { qos: 1 }, async (topic, payload) => {
        const result = await handler(payload);
        const execId = /* 从 topic 提取 */;
        await this.client.publish(
          `$emqx/commands/things/${this.thingName}/executions/${execId}/response`,
          { status: "SUCCEEDED", result }
        );
      }
    );
  }

  // === Delta 监听 ===
  onDelta(handler) {
    this.client.subscribe(
      `$emqx/things/${this.thingName}/shadow/update/delta`,
      { qos: 1 }, handler
    );
  }
}
SDK 兼容性： 生成的 SDK 同时支持 MQTT 和 HTTPS（ThingDatas API）两种接入方式。 SDK 内部自动处理 Shadow 三态同步、命令 request/response 匹配、Jobs 协议循环。

🤖 DA 现有功能适配 Fleets 协议

设计要点： DA 的 AI 功能（NL 控制、规则引擎、数据分析）在控制设备时，通过 Fleets REST API 下发控制指令，而非直连 MQTT。 设备本身仍走 $emqx/ 协议（由 Fleets 代理下发），完全无感知 —— 它只看到 Fleets 同一套协议。

🔄 数据流 · 上行共享 + 下行独立

核心原则： 上行设备上报经 EMQX Rule Engine 落入 Fleets（持久化），DA 通过 Fleets 数据/事件 API 获取，不再直连 MQTT； 下行DA 经 Fleets API 下发控制指令，由 Fleets 统一经 MQTT 送达设备。Fleets 是设备控制的唯一权威入口。

📊 完整数据流对照

数据类型 方向 MQTT 主题 Fleets 处理 DA 处理

Events↑$emqx/things/{name}/events/{type}Rule Bridge → EMQX Tables经 Fleets 事件 API 触发
Command response↑$emqx/commands/.../{id}/responseRule HTTP POST → ThingDatas API → PG经 Fleets API 回调处理

Command request↓$emqx/commands/.../{id}/requestEMQX V5 API 发布经 Fleets API 下发
Jobs notify↓$emqx/things/{name}/jobs/notifyEMQX V5 API 发布—（DA 不做 Jobs）
Lifecycle⚡$events/client_connected|disconnectedRule HTTP POST → ThingDatas API → PG经 Fleets 生命周期事件 API

🎮 统一经 Fleets 的设备控制

核心设计： DA 和 Fleets 都通过 Fleets 下发设备控制：Fleets 管理 Shadow 三态同步和 Jobs 调度，DA 的 AI 会话式控制与规则引擎触发也统一经 Fleets API 下发。 Fleets 是设备控制的唯一权威入口，DA 不再直连 MQTT，因此不存在两端并发写入的冲突。

🎯 DA 控制场景

"把客厅温度调到 26°C" → DA AI Agent 解析意图 → 经 Fleets API 下发 Shadow desired 或 Command → Fleets 经 MQTT 送达设备执行。

🏭 Fleets 控制场景

批次重启 10 台设备 → Fleets 创建 Job → EMQX V5 API 发布到 $emqx/ 主题 → 设备收到后执行并上报。

⚡ 控制权威

DA 与 Fleets 的控制指令都经由 Fleets 下发，Fleets 是设备控制的唯一权威入口。DA 不再直连 MQTT，因此不会产生与 Fleets 并发写入的冲突。

📋 一致性与状态

设备状态以 Fleets Shadow 为权威存储，DA 通过 Fleets API 读取并缓存 thingType 模型，控制指令经 Fleets 统一落库与下发。

🔌 FleetsAdapter · Device Agent 适配层

适配层职责： FleetsAdapter 是 DA 中的一个内部模块，负责拉取 Fleets ThingType Schema、构建 DeviceSpec，并封装经 Fleets API 的设备控制调用，使得 SDK 生成器和 AI 功能能正确适配 Fleets 协议。 它不直连 MQTT，也不处理设备消息与 AI 逻辑本身。

🏠 DA 原生协议作为一个平台规范

核心设计： DA 的原生协议也是一个平台规范，与 Fleets 同等待遇。DA 初始化时指定 platform: "native" 则加载原生平台规范库、使用 DA 自有主题与设备通信。 指定 platform: "fleets" 则加载 Fleets 规范库、使用 $emqx/ 主题通信。两者是 独立的 DA 实例，不是同一实例的多模式。
关键变化： DA 原生协议不是另一个 Adapter 在同一个实例里切换，而是 另一个 DA 实例。 每个 DA 实例在初始化后就锁定了平台规范，运行时不变。

🔌 跨平台抽象：单实例单平台

Device Agent 初始化时指定一个平台规范（如 Fleets），根据对应平台库生成物模型和 SDK。 每个 DA 实例运行一套平台规范，多个平台用多个 DA 实例。DA 原有功能通过平台规范库适配该平台与设备的交互。
核心原则： DA 启动时通过 platform: "fleets" 配置锁定目标平台。 平台规范库提供该平台的物模型映射、MQTT 主题模板、Payload 格式定义。 DA 的 AI Core 不感知具体平台 —— 它看到的始终是经过平台库翻译后的标准化交互接口，但 每个 DA 实例只绑定一个平台。

⛓️ 保证机制详解（新方案）

1. DA AI Core 不变 — NL 控制器、规则引擎、AI 分析器、SDK 生成器始终操作规范化的 DeviceSpec。AI Core 代码零改动。

2. 平台规范库在初始化时生成一切 — DA 启动时指定 platform: "fleets"，平台库拉取 ThingType Schema、生成 DeviceSpec 和 MQTT 主题模板、输出 SDK。这个过程中确定的所有交互规范在运行时不变。

3. 单实例单平台，无路由开销 — 每个 DA 实例只处理一种平台的设备。所有交互通过该平台的规范运行。不需要在运行时判断设备类型、不需要 Adapter Router。

4. 新增平台 = 新增规范库 + 新 DA 实例 — 实现该平台的规范库（PlatformSpec 接口），Fleets 启动对应的新 DA 实例并赋值 platform 即可。不影响已有实例。

// DA 启动配置

// DA 初始化流程
async function initDA(config) {
  // 1. 根据 platform 加载对应规范库
  const specLib = loadPlatformSpec(config.platform);
  // → @da/platform-fleets 或 @da/platform-native

  // 2. 规范库初始化
  await specLib.init(config);

  // 3. 生成设备物模型
  const deviceSpec = await specLib.fetchThingType("thermostat-v1");
  // → 包含 properties, events, commands, mqttTopics

  // 4. 生成设备 SDK（如需要）
  const sdk = specLib.generateSdk(deviceSpec, "typescript");

  // 5. 注入 DA AI Core
  const daCore = new DACore(specLib);

⚙️ 控制面总览：Fleets 管控 DA 生命周期，DA 按需拉取设备模型

Fleets 作为设备管理平台，对 Device Agent 拥有控制面权限：管理 DA 实例的生命周期（进程级）。 设备模型采用「按需拉取」模式 —— DA 不保存完整设备列表，运行时主动用 thingId 调 Fleets API 拉取该设备的 thing metadata，从中提取并只缓存其 thingType（即 DeviceSpec）。
两层控制面设计： ① 实例层 — Fleets 管理 DA 进程生命周期（启动/停止/重启/健康监控） ② 控制层 — DA 对 Fleets 设备拥有完整的 NL 控制/规则引擎/AI 分析权限，交互所需的 DeviceSpec 由 DA 按 thingId 从 Fleets 拉取 thing metadata 后提取其 thingType
无需设备导入/导出： DA 不再维护设备列表，也不提供导入/导出能力。 仅在与某设备交互时按 thingId 调 Fleets API 拉取其 thing metadata，提取并只缓存 thingType（DeviceSpec），再通过 Fleets API 间接下发控制。
DA API 认证： Device Agent 的 HTTP API 计划支持 Basic Auth 认证。 用户或外部系统可直接使用 Basic Auth 调用 DA API，无需经过 Fleets 转发。 Fleets 仍可保留通过服务间凭证调用 DA 控制面 API 的能力。

🔄 DA 实例生命周期管理（进程控制）

方式： Fleets 通过进程控制方式管理 DA 实例的生命周期。DA 作为受管进程，在 Fleets 注册并发送心跳。 Fleets 通过操作系统的进程管理机制（systemd / launchd / Docker）或 SSH 远程控制 DA 实例的启停。

📦 设备模型获取：按 thingId 拉取 metadata（只存 thingType）

核心逻辑： DA 不保存完整设备列表。当 DA 需要与某个设备交互时（NL 控制、规则触发、AI 分析）， 主动用 thingId 调 Fleets API 拉取该设备的 thing metadata，从中提取并只缓存其 thingType（即 DeviceSpec），同类型设备共享一份模型，随后通过 Fleets API 下发控制指令（Fleets 再经自身 MQTT 通道与设备通信）。

🟢 默认流程：按 thingId 拉取 metadata（只存 thingType）

// DA 需要与设备交互时（示例：NL 控制）
async function resolveDeviceSpec(thingId) {
  // 1. 按 thingId 拉取 thing metadata，查该设备属于哪个 ThingType
  const thing = await fleetsApi.get(`/api/v1/things/${thingId}`);
  const typeName = thing.thingTypeName;

  // 2. ThingType 缓存命中则直接用（同类型共享）
  if (specCache.has(typeName)) return specCache.get(typeName);

  // 3. 未命中 → 拉取 ThingType Schema 构建 DeviceSpec
  const thingType = await fleetsApi.get(
    `/api/v1/thing-types/${typeName}`
  );
  const spec = buildDeviceSpec(thingType);

  // 4. 只缓存 thingType（DeviceSpec）模型，不保存设备列表
  specCache.set(typeName, spec);
  return spec;
}

// 交互时无需预先导入设备（DA 不维护设备列表，按 thingId 经 Fleets API 控制）
async function handleNlControl(input, thingId) {
  const spec = await resolveDeviceSpec(thingId);
  const cmd  = parseIntent(input, spec.commands);
  // 控制指令统一经 Fleets API 下发，Fleets 再经 MQTT 送达设备
  await fleetsApi.post(`/api/v1/things/${thingId}/commands`, {
    command: cmd.name,
    params:  cmd.params
  });
}

🤖 DA 对 Fleets 已有设备的控制权

核心机制： DA 按 thingId 拉取到设备的 thing metadata、提取其 thingType（DeviceSpec）后，即通过 Fleets API 下发控制指令（Fleets 再经 MQTT 与设备通信）， 控制路径统一经过 Fleets，DA 不再直连 MQTT Broker。设备模型按需获取，无需预先导入。

🎯 DA 获得哪些控制权

• 通过 Fleets API 读取设备上报的 Shadow reported、Events、Command response
• 通过 Fleets API 设置 Shadow desired 属性
• 通过 Fleets API 下发 Command 指令
• 通过规则引擎自动响应设备事件
• 通过 AI 分析设备数据趋势

🔒 Fleets 保留哪些控制权

• 设备注册与注销（Thing CRUD）
• ThingType Schema 的创建/修改
• Jobs 作业调度（DA 不做 Jobs）
• 设备认证凭据管理
• 与设备之间的 MQTT 通道（Fleets 代理下发）

⚡ 按需获取模型

DA 不维护设备列表，运行时用 thingId 向 Fleets 拉取 thing metadata，提取其 thingType 构建 DeviceSpec 并只缓存该模型。 DA 的控制指令统一经 Fleets 下发，Fleets 是设备控制的唯一权威入口；DA 不再直连 MQTT。
